## 1. Test Scope

**In scope:**
- Verify that the final assembled HTML (`trip_full_LANG.html`) produced by the parallelized render pipeline is structurally identical to what the sequential pipeline would produce
- Confirm that all existing regression tests pass without modification after the parallelization change
- Validate that day section ordering, count, and completeness are preserved

**Out of scope:**
- Testing the parallelization orchestration itself (batch assignment logic, subagent spawning, retry logic) — these are LLM workflow rules in `rendering-config.md` and `SKILL.md`, not application code. They cannot be asserted via Playwright against the HTML output.
- Performance benchmarking (REQ-002 AC-4: wall-clock time reduction) — this is an empirical observation during trip generation, not an automated test assertion.
- Testing incremental rebuild mode (REQ-007) — this is a workflow path, not an HTML output difference. Existing regression tests cover the HTML correctness regardless of which mode produced it.
- New POM locators — no new HTML structure is introduced.

**Test type:** Regression (existing suite serves as the validation gate)

**Key insight:** This change modifies *how* HTML fragments are generated (parallel vs sequential) but not *what* is generated. The HTML output contract is unchanged. Therefore, the existing regression test suite (`structure.spec.ts`, `day-cards.spec.ts`, `poi-cards.spec.ts`, `navigation.spec.ts`, `activity-poi-linking.spec.ts`, `overview-budget.spec.ts`, etc.) already validates every BRD acceptance criterion that manifests in the HTML output. The test strategy is: **run the full existing regression suite against HTML produced by the new parallel pipeline and confirm zero regressions.**

## 3. Test Cases

### TC-001: All day sections present and correctly ordered

**Traces to:** REQ-001 → AC-1, AC-2, AC-3; REQ-005 → AC-1; REQ-006 → AC-5
**Type:** Regression (existing)
**Spec file:** `structure.spec.ts`
**Priority:** Critical

**Preconditions:**
- `trip_full_LANG.html` generated using the new parallel pipeline

**Steps:**
1. Load the HTML page
2. Count all `.day-card[id^="day-"]` elements
3. Verify each day section `#day-0` through `#day-N` exists

**Expected result:**
- Day section count equals `tripConfig.dayCount`
- All day IDs are present in sequential order (no gaps from batch boundaries)

**Implementation notes:**
- Already implemented in `structure.spec.ts` → "should render all day sections"
- Uses shared-page fixture; no changes needed
- This test inherently validates that batch assignment produced all days with no gaps or overlaps, because the HTML is the final assembled output

## 6. Risk & Mitigation

| Risk | Mitigation |
|---|---|
| Fragment ordering error at batch boundaries (e.g., batch 2 days appear before batch 1 days) | TC-001 validates sequential day ordering 0..N in the assembled HTML |
| A subagent omits a component (e.g., missing Plan B section in one day) due to incomplete prompt contract | TC-002 checks every structural component per day with soft assertions — pinpoints exact day and component |
| Fragment file contains wrapper elements (`<html>`, `<style>`) causing duplicate tags in assembled HTML | TC-006 validates single inline `<style>` tag and no external links; `structure.spec.ts` would detect malformed DOM |
| Visual regression baselines become stale if any unrelated CSS change is included | Keep visual baselines version-controlled; review diffs before accepting updates |
| Cross-reference breakage: activity label in day X links to POI in same day, but fragment was generated by different subagent batch | TC-003 validates all activity-label-to-POI links resolve correctly across the full document |
